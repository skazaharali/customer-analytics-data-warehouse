# Star Schema Design

## 1. Core Design Decisions (The "Why")

### Why Separate Fact Tables?
A common anti-pattern is attempting to flatten payments and refunds into `fact_orders`. We explicitly model `fact_payments` and `fact_refunds` as separate fact tables to preserve the grain. 
*   **One order** can have **multiple payment attempts** (e.g., a failed credit card followed by a successful one).
*   **One order** can have **multiple partial refunds**.
*   Capturing these inside `fact_orders` would cause a fan-out, destroying the order-level grain and inflating revenue metrics.

### Key Terminology
*   **Surrogate Key (`_sk`):** An auto-generated warehouse key used to uniquely identify a specific historical version of a dimension (e.g., `customer_sk`).
*   **Business Key (`_id`):** The immutable, natural identifier generated by the source system (e.g., `customer_id`, `payment_id`).

## 2. Dimensions

### `dim_customer_scd`
*   **Purpose:** Stores customer history and tracks status changes over time.
*   **Grain:** One row per customer version.
*   **Business Key:** `customer_id`
*   **Surrogate Key:** `customer_sk` (Primary Key)
*   **SCD Type:** Type 2

**Attributes:**
*   `customer_sk` 
*   `customer_id` 
*   `customer_name`
*   `email`
*   `city`
*   `status`
*   `start_date`
*   `end_date`
*   `is_current`

### `dim_date`
*   **Purpose:** Shared calendar analysis.
*   **Grain:** One row per calendar date.
*   **Primary Key:** `date_key`

**Date Mappings (How Facts Connect):**
*   `fact_orders.date_key` = `order_date`
*   `fact_payments.date_key` = `payment_date`
*   `fact_refunds.date_key` = `refund_date`
*   `fact_events.date_key` = `event_date`

## 3. Facts

### `fact_orders`
*   **Purpose:** Records every completed purchase.
*   **Grain:** Exactly 1 row per order.
*   **Business Key:** `order_id`
*   **Foreign Keys:** `customer_sk`, `date_key`
*   **Measures:** `order_amount`, `shipping_amount`, `discount`, `tax`
*   **Partition Strategy:** `order_date` (Supports incremental recomputation)
*   **Refresh Strategy:** Incremental | MERGE | Late-arriving data supported | Idempotent

### `fact_payments`
*   **Purpose:** Records individual payment attempts.
*   **Grain:** 1 row per payment_id.
*   **Business Key:** `payment_id`
*   **Foreign Keys:** `customer_sk`, `date_key`, `order_id` (FK to business transaction)
*   **Measures:** `paid_amount`
*   **Partition Strategy:** `payment_date`
*   **Refresh Strategy:** Incremental | MERGE | Late-arriving data supported | Idempotent

### `fact_refunds`
*   **Purpose:** Records individual refund events.
*   **Grain:** 1 row per refund_id.
*   **Business Key:** `refund_id`
*   **Foreign Keys:** `customer_sk`, `date_key`, `order_id` (FK to business transaction)
*   **Measures:** `refund_amount`
*   **Partition Strategy:** `refund_date`
*   **Refresh Strategy:** Incremental | MERGE | Late-arriving data supported | Idempotent

### `fact_events`
*   **Purpose:** Captures website activity and behavioral events.
*   **Grain:** 1 row per event.
*   **Business Key:** `event_id`
*   **Foreign Keys:** `customer_sk`, `date_key`
*   **Measures:** Primarily additive counts. Optional metrics: `event_duration`, `session_length`, `scroll_depth`.
*   **Partition Strategy:** `event_date`
*   **Refresh Strategy:** Incremental | MERGE | Late-arriving data supported | Idempotent
